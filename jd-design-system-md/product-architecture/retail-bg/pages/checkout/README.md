# 结算页 · Checkout

> 用户从"想买"到"下单"的关键一步。**结算页的核心矛盾是"展示完整 vs 决策成本低"**——价格构成要透明,但不能让用户算糊涂。

---

## 模块清单

```
结算页
├── [Module] 顶部导航("提交订单"标题)
├── [Module] 收货地址(必选)
├── [Module] 商品列表(简化 CheckoutItem,不可编辑数量)
├── [Module] 配送方式(标准 / 当日 / 次日 / 自提)
├── [Module] 优惠券 / 京豆 / 礼品卡(可选叠加)
├── [Module] 发票
├── [Module] 备注(可选)
├── [Module] 价格明细(商品总额 + 运费 - 优惠 = 实付)
└── [Bottom Bar] "提交订单"主 CTA
```

---

## 收货地址(关键模块)

```
┌─────────────────────────────────┐
│ 张三 138 1234 5678              │
│ 北京市朝阳区... 详细地址        │
│                          [更换]  │
└─────────────────────────────────┘
```

- 默认地址自动填入
- 点击 / "更换"→ 跳转地址列表
- 无地址 → "请添加收货地址"+ 主 CTA disabled

---

## 价格明细

```
商品总额          ¥1,234.00
运费                  ¥0.00
优惠券              -¥30.00
京豆抵扣             -¥5.00
─────────────────
应付总额        ¥1,199.00
```

- 主价用 `Price.Large` + 京东红
- 减项用 `text.danger`(显示负值)
- 加项用 `text.primary`

---

## 优惠叠加规则

- 优惠券:跳转选择优惠券 sheet
- 京豆:checkbox + 抵扣金额(实时算)
- 礼品卡:可选输入卡号
- **铁律**:优惠规则要透明,不能"看不出怎么算的"

---

## 底部固定栏

```
┌─────────────────────────────────┐
│ 应付:¥1,199.00       [提交订单]  │
└─────────────────────────────────┘
```

- 主 CTA = primary block + L size
- "提交订单" loading 状态拦截重复点击
- 提交成功 → 跳转支付页

---

## 关键 KPI

| KPI | 当前 | 目标 |
|---|---|---|
| 结算 → 下单转化率 | - | ↑ 0.5% |
| 优惠券使用率 | - | - |
| 提交失败率 | - | ↓ |

---

## 状态完整性

- ✅ 默认
- ✅ 无地址(主 CTA disabled + 引导添加地址)
- ✅ 库存变化(部分商品下架/限购)→ 提示用户去购物车更新
- ✅ 价格变化 → 提示更新
- ✅ 提交失败 → 错误 + 重试

---

## 反例

| ❌ 反面 | 解释 |
|---|---|
| 提交按钮 loading 不拦截重复点击 | 重复下单(2024-12 事故) |
| 价格明细不可见(藏在折叠里) | 用户不信任 |
| 优惠叠加算不清(默认勾选难取消) | 用户感觉被坑 |
| 收货地址可滑动覆盖到价格区 | 主信息被挡 |
| 提交按钮被键盘挡 | 弹键盘时滚动 |
