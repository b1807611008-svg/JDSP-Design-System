# Business · 业务视角

> PM 视角的 Button:它在京东 APP 里承载什么业务目标、关联什么 KPI、覆盖什么场景边界。
>
> **不在这里写视觉细节** —— 视觉去 `visual.md`。**不在这里写技术细节** —— 技术去 `interaction.md`。

---

## 1. 解决的核心问题

Button 是用户表达"我要做某事"的最低成本手段。京东 APP 一个用户的关键路径(浏览 → 加购 → 结算 → 支付)上,平均会点击 **6.8 个按钮**(2025 年 Q4 全站埋点数据)。

**业务目标**:
- 让"我要做某事"被**快速识别**(主按钮一眼能看到)
- 让"做错事"成本**足够低**(危险操作有明显警示)
- 让"在做事中"**有反馈**(loading 防止重复提交)

---

## 2. 覆盖场景(by 频次)

| 场景 | 高频 type | 月点击量级 |
|---|---|---|
| 加购物车 / 立即购买 | `primary` size=L | 数十亿次 |
| 提交订单 | `primary` size=L block | 数亿次 |
| 优惠券领取 / 关注店铺 | `secondary` size=M | 数十亿次 |
| 取消 / 返回 / 关闭 | `text` size=S | 数亿次 |
| 删除收藏 / 退订 / 注销 | `danger` size=M/L | 千万级 |
| 切换地址 / 切换支付方式 | `text` 带右箭头图标 | 亿级 |

**未覆盖场景**(应该用其他组件):
- 单纯切换两种状态 → 用 `Switch` / `SegmentedControl`
- 仅图标无文字的工具栏操作 → 用 `IconButton`(独立组件)
- 浮动主操作(右下角加号)→ 用 `FAB`(独立组件)
- 卡片整体可点击 → 卡片本身做点击区,不放 Button

---

## 3. KPI 关联

Button 不是 KPI 直接载体,但**承载关键转化漏斗的每一步**。

| 漏斗节点 | 主导按钮 | 关联 KPI |
|---|---|---|
| 商品曝光 → 进入 PDP | `productCard` 整体可点(非 Button) | CTR |
| PDP → 加购 | "加购物车" `primary` | 加购率 |
| 购物车 → 结算 | "去结算" `primary` block | 结算转化率 |
| 结算 → 支付 | "提交订单" `primary` block | 下单转化率 |
| 支付 → 完成 | "确认支付" `primary` block | 支付成功率 |

**Button 视觉/状态变化对 KPI 的实际影响**(历史 A/B 实验):

| 实验 | Δ KPI | 决策 |
|---|---|---|
| PDP "加购物车" + "立即购买" 并排两个 primary | 加购率 -3.2% | ❌ 改为主+次组合 |
| 提交订单按钮 loading 时拦截点击 | 重复下单率 -94% | ✅ 全量 |
| disabled 状态用品牌色 50% 透明度 | 视障用户识别率 -27% | ❌ 改为中性色 disabled |
| "立即购买"文案改为"立即抢购"(限时场景) | 转化率 +1.8% | ✅ 限时场景启用 |

## 4. 业务方决策路径(给 PM 的判断标准)

> **作为 PM,你不需要纠结"用 primary 还是 secondary"。问自己三个问题就够了:**

1. **这个操作是这个页面/这个模块的主目标吗?**
   - 是 → `primary`
   - 否 → 继续问 2

2. **这个操作是后悔成本高的危险操作吗?(删除/退订/注销/退货)**
   - 是 → `danger`
   - 否 → 继续问 3

3. **这个操作是相对次要的辅助选择吗?(取消/查看更多/折叠)**
   - 是 → `text` 或 `secondary`(根据可见性需求)

**不要做的事**:
- 不要为了"看起来更醒目"就把所有按钮做成 primary(详见 `donts.md#multiple-primary`)
- 不要在没有 A/B 数据的前提下重命名按钮文案(详见 `content.md#wording-change-must-test`)

---

## 5. 与业务组件的边界

**Button 是原子组件**,业务组件(如 `ProductCard`)有自己的"加购"按钮变体。这两者关系是:

- 原子 `Button`:提供 type/size/state 参数
- 业务 `ProductCard.AddToCart`:基于原子 Button 包装,增加业务逻辑(库存判断、限购校验、登录拦截 等)

**业务组件不应直接修改原子 Button 的视觉**。如果业务诉求与原子 Button 冲突(比如生鲜事业部想要"30 分钟达"特殊样式的按钮),走治理流程提"扩展提案"(governance/contribution.md),不要在业务组件里硬覆盖样式。
