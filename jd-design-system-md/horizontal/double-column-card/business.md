# 业务背景与五大家族 · Business

> 给业务线设计师 / 老板看：京东双列卡为什么是「五大家族」体系，每个家族的业务诉求是什么。
>
> **执行细节**（22 个基线完整字段、采集样本、家族决策树伪代码）见 Skill 仓库 [`references/business-baselines.md`](https://github.com/ShuaiMXu/jd-double-column-card-skill/blob/main/references/business-baselines.md)。

---

## 一、为什么是「五大家族」而不是二元

最初的直觉是「内容 vs 商品」二元区分——但实战采集 22 个基线后发现根本不够：

| 误区 | 真相 |
| --- | --- |
| 二元（内容 / 商品） | **五元**（内容 / 商品 / 圈子 / 点评 / 新品）|
| 家族由封面决定 | **家族由底部不变量决定** |
| Tab 名称推断业务线 | **business_line 必须人工声明**（Tab 命名易变）|
| 一种规则套所有卡 | **L1 跨家族 + L2 家族专属** 双层 |

「家族」不是分类便利，是**用户在站内跨业务线跳转时的识别心智单元**。

## 二、五大家族体系详解

### 2.1 内容家族（Content）

**心理锚点**：UGC 信任 + 社交证明
**底部 DNA**：头像 + 昵称 + 互动数据
**主语**：创作者（人）

**业务诉求**：让用户相信「这是真人推荐的」，激发种草欲。

**典型业务线**：推荐流（种草 / 测评）、JoyAI、达人内容

**已采子类型**（共 7）：
- 图文卡（点赞 / 观看互动两种）
- 直播卡（基础 / 红包雨变体 / 超头变体 / 预约变体 / 沉浸式变体）

### 2.2 商品家族（Product）

**心理锚点**：商家背书 + 销量证明
**底部 DNA**：店铺 logo + 店铺名 + 销售数据
**主语**：商家

**业务诉求**：让用户产生交易决策——价格、销量、店铺信誉是核心。

**典型业务线**：主站 / 京东超市 / 生鲜

**已采子类型**（共 5）：视频卡 / 直播卡（基础 / 弹幕外露 / 沉浸式商卡 / 沉浸式促销）

> ⚠️ **L2 待补**：商品家族 L2 规则尚未完整定义，当前 Skill 对商品家族卡只走 L1 跨家族规约。

### 2.3 圈子家族（Community）

**心理锚点**：集体归属感
**底部 DNA**：圈子图标 + 圈子名（**无作者、无互动数据**）
**主语**：社群

**业务诉求**：传达「这是一群人在一起讨论」，用社群感降低个体决策压力。

**典型业务线**：京东养车、美妆社区

**已采子类型**：图文卡（视频卡按规律预测，未采集）

### 2.4 点评家族（Review）

**心理锚点**：UGC 信任 + 决策证明
**底部 DNA**：头像 + 昵称 + 互动数据 + **标题下种草数**
**主语**：推荐者（人）+ 决策证明

**业务诉求**：在内容信任之上**叠加决策辅助**——「不只是种草，还是已经被验证过的种草」。

**典型业务线**：点评 / Beauty / 厨房

**已采子类型**（共 4）：图文卡 / 视频卡 / 种草混血 ×2

> ⚠️ **关键陷阱**：点评家族和内容家族**底部基础结构相同**（都是头像+昵称+互动），区分必须靠 `business_line` 辅助，不能只看底部。详见 [`donts.md`](./donts.md) 的「不要把动态数据当家族标识」。

### 2.5 新品家族（NewProduct）

**心理锚点**：事件 / 活动 / 趋势驱动
**底部 DNA**：按子类型变化（CTA 按钮 / 事件胶囊 / 商品挂载 / 话题标签等）
**主语**：**抹除主语，聚焦事件本身**

**业务诉求**：通过场景叙事而非个体推荐，营造「现在是新事件发生的时刻」的紧迫感。

**典型业务线**：新品频道

**已采子类型**（共 5，对应购买决策 5 阶段）：
- 入口卡（造势）
- 直播入口（参与）
- 测评视频（验证）
- 趋势卡（探索）
- 明星同款（共鸣）

> 💡 **设计哲学洞察**：「**新品家族 5 子类型 = 购买决策 5 阶段**」——造势 → 参与 → 验证 → 探索 → 共鸣。这种"用家族子类型映射用户决策阶段"的思路是其他家族也可以借鉴的。

## 三、设计哲学：「顶部+中部统一识别语言；底部锚定家族身份」

```
卡片视觉解剖
─────────────────────────
顶部     ←── 跨家族共享识别语言（属性表达位、辅助信号位、元信息位）
中部     ←── 跨家族共享识别语言（封面、标题、副标题）
─────────────────────────
底部     ←── 家族 DNA 差异（业务线表达独特叙事）
```

**效果**：
- 用户跨业务线跳转：顶部+中部识别心智可复用 → **统一 App 心智**
- 业务线自由表达：底部保留差异 → **业务线独特叙事**

这是京东作为多业务线超级平台的核心设计哲学，而不仅仅是双列卡的局部规约。

## 四、双标签协议（关键工程细节）

| 标签 | 颗粒度 | 来源 | 枚举 |
| --- | --- | --- | --- |
| `business_line` | 页面 / 流级 | **人工显式声明** | `推荐 \| 点评 \| 圈子 \| 新品 \| 个人中心 \| 直播` |
| `card_type` | 单卡级 | Skill 视觉识别（VLM 天职）| `内容-图文 / 内容-直播 / 商品-视频 / 圈子-图文 / 点评-图文 / 新品-入口 / ...` |

**为什么分两层**：
- 页面内 tab 命名会变（JoyAI 改名求真官、求真官又可能改名 ZZZ）—— 用可变信号做业务分类不稳定
- 把易变环节（business_line）交给人工输入，让 Skill 保持稳定

**协议铁律**：缺失 business_line → **拒绝执行，不推断**。

## 五、家族识别的判断顺序（人能理解的描述）

家族识别按「**底部 → 封面 → 左上角**」的顺序——底部不变量决定家族归属，封面只做装饰。判断的两个关键约束：

1. **内容家族 ≠ 点评家族（底部基础结构相同）**：必须用 `business_line` 辅助区分（参见 [`donts.md`](./donts.md) 第 2 条）
2. **动态数据降级**：种草数、观看量、身份标签等是激活信号，不参与家族判定（参见 [`experience.md`](./experience.md) 原则 8）

> ⚠️ **可执行决策树（Step 0-6 伪代码 + IF/ELSE 分支）属执行层，留在 Skill 仓库**：[`prompts/page-level-review.md`](https://github.com/ShuaiMXu/jd-double-column-card-skill/blob/main/prompts/page-level-review.md) Step 2a。本文件不复制——避免 Wiki 和 Skill 双向同步。
