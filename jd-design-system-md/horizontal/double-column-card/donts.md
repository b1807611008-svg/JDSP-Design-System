# 不要做的事 · Don'ts

> Skill 维护者 / 调用方必读。这里的每一条都是**实战驱动的血教训**，不是理论推演。
>
> 详细演化历程见 [`CHANGELOG.md`](./CHANGELOG.md) 和 Skill 仓库 [`references/changelog.md`](https://github.com/ShuaiMXu/jd-double-column-card-skill/blob/main/references/changelog.md)。

---

## ❌ 1. 不要让 Skill 推断 `business_line`

**反例**：从页面 URL / Tab 名称 / 当前栏目猜业务线归属。

**为什么**：京东内 tab 命名易变（JoyAI → 求真官 → ZZZ），用可变信号做业务分类，结果会跟随平台改名漂移。

**正确做法**：调用方**必须人工显式声明** `business_line`，缺失则 Skill **拒绝执行**，不推断。

> 来源：v0.2 实战教训。Shaka 协作指示「页面内 tab 命名易变，用可变信号做分类不稳定」。

---

## ❌ 2. 不要把动态数据当家族识别硬条件 ★ v0.5.2 血教训

**反例**：v0.5.1 决策树写 `IF 标题下有"X 万人种草" → 点评，ELSE → 内容`。

**为什么**：种草数是**运行时数据**，0 人种草就不显示。用「有无种草数」识别家族 → 错把"未激活的点评卡"判成"内容家族卡"。

**正确做法**：
- 家族识别靠**结构不变量**（固定元素的存在与位置）
- 动态数据（种草数 / 观看量 / 身份标签）作为**激活信号**记录在 annotations，**不参与家族判定**
- 内容家族和点评家族底部基础结构相同 → **必须用 `business_line` 辅助区分**

> 来源：v0.5.2 实战测试驱动的原则级修订。详见 [`experience.md`](./experience.md) 原则 8。

## ❌ 3. 不要把用户/运营上传的内容层纳入观察 ★ v0.5.2 血教训

**反例**：v0.5.1 报告写 `Card X 酒店图片偏横 → 瀑布流节奏不齐`。

**为什么**：封面内容是用户/运营上传，**设计师无法强控制**。规范只能管「框」，不能管「画里的内容」。如果把内容层纳入观察，报告会充斥「这张图比例偏横」这种**无法修复的伪差异**。

**正确做法**：
- ✅ 观察：框体尺寸、圆角、间距、字号 token、组件有无、元素位置
- ❌ 不观察：封面图具体内容、构图、比例偏向、标题文案质量、头像图像、运营配图色彩
- ⚠️ **例外**：图片比例**明显超出规范值域**（如 1:2 极端比例）属规范违规，要报；同一值域内不同选择（3:4 vs 4:3）不属于差异

> 来源：v0.5.2 实战测试驱动的原则级修订。详见 [`experience.md`](./experience.md) 原则 7。

## ❌ 4. 不要做比规范更细的判定（Skill 不下钻）

**反例**：业务方对「梯度叠加封面每层」「明星同款的具体趋势元素」没有规范，Skill 自己定义判定细则。

**为什么**：Skill 的权威性来自规范，规范模糊处 Skill 自定就是僭越。**Skill 的判定深度应永远 ≤ 规范的明确深度**。

**正确做法**：
- 描述性识别（"这是梯度叠加封面" / "这是明星同款变体"）
- 不强制规约（"不深挖每层规则" / "不评价具体趋势元素"）
- 业务方补规范后，Skill 同步深化

> 来源：原则 6「识别深度匹配规范深度」。

## ❌ 5. 不要用「违规判决」语言，要用「差异观察」语言

**反例**：报告写 `P0 违规：标题字号偏大；P1 违规：互动数据缺失`。

**为什么**：Skill 不是评审权威，最终裁决权在设计师 + 评审委员会。判决式语言会引发反弹（设计师感觉被指控），同时漏掉业务合理性。

**正确做法**：
- 每条差异 4 字段：差异描述 + 体验视角评价 + **可能的业务合理理由** + 讨论建议
- 输出形态是**讨论抓手**，不是判决书
- 「这里不一样；可能有 X 体验问题，也可能有 Y 业务合理性——请讨论」

> 来源：v0.3.0 定位调整。Shaka 协作指示「你发现不一致的地方不一定要纠错，你会给报告，告诉我们哪些不一样的，并用最开始教你的体验设计的视角去评价」。

## ❌ 6. 不要在每条差异里堆砌多个体验维度

**反例**：一条差异同时讲「认知负担 + 视觉一致性 + 信息层级 + 行动号召」，最后看不出哪个是核心。

**为什么**：[`experience.md`](./experience.md) 维度工具箱有 10 个维度，目的是**让你挑一个最贴切的讲透**，不是让你都堆上去。

**正确做法**：每条差异只挑一个维度讲透；维度的选择本身体现判断质量。

> 来源：v0.3.1 报告精简纪律。

---

## ❌ 7. 不要让报告超过一屏

**反例**：八段式报告 + 10 维评价工具箱 + 完整附录，看完报告需要滚动十屏。

**为什么**：报告的目的是**推动讨论**。如果设计师/评审看不完，报告就是死物。

**正确做法**（v0.3.1 输出纪律）：
- 三段式：整体印象 + 差异表 + 下一步建议
- 差异表最多 8 行
- 每格限字（差异 ≤30 字 / 体验代价 ≤40 字 / 业务理由 ≤25 字 / 建议 ≤30 字）
- 整体印象最多 3 句
- 目标：**一屏能看完**

> 来源：Shaka 反馈「报告简单点，刚看了一下很复杂呢」。

## 元规则：什么时候反思 don't 清单本身

下列信号说明 Skill 当前 don't 清单可能不够 / 需要补充：

- 同一类错误**重复出现** → 升级为新 don't
- 调用方**反复问同一个问题** → 该问题可能需要 don't 解释
- 实战测试**发现新原则级问题** → 走 changelog v0.x.x 修订流程

新 don't 加入流程：
1. 在 Skill 仓库 `references/changelog.md` 记录触发场景
2. 同步更新本文件 + 对应 [`experience.md`](./experience.md) 原则
3. 如涉及视觉规约或元素位置 → 同步更新 [`visual.md`](./visual.md)
4. 如涉及输入输出协议或触发词 → 同步更新 [`ai-schema.md`](./ai-schema.md)
5. 同步更新 Skill 仓库 `prompts/page-level-review.md` 执行约束
6. 在 [`CHANGELOG.md`](./CHANGELOG.md) 记录版本演进
