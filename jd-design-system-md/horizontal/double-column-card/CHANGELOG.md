## v0.5.2 (2026-04-14) · 实战血教训驱动的两个原则级修订 ⭐

> Shaka 用一张点评场域页面截图做实战测试，暴露 v0.5.1 两个原则级问题——**非小补丁，是基础原则调整**。

### 新增原则 7：结构优先、内容无涉

**触发场景**：v0.5.1 报告写「Card X 酒店图片偏横 → 瀑布流节奏不齐」。

**问题**：封面内容是用户/运营上传，设计师无法强控制。规范只能管「框」，不能管「画里的内容」。

**修正**：
- 明确 Skill 观察范围 = 设计师规范可控的结构性元素
- 不观察 = 用户/运营上传的内容层
- 例外：图片比例超出规范值域属规范违规要报

详见 [`donts.md`](./donts.md) 第 3 条 + [`experience.md`](./experience.md) 原则 7。

### 新增原则 8：动态数据不作家族识别硬条件

**触发场景**：v0.5.1 决策树写 `IF 标题下有"X 万人种草" → 点评`，结果错把"未激活的点评卡"判成"内容家族卡"。

**问题**：种草数是运行时数据，0 人种草就不显示，不是结构标识。

**修正**：
- 家族识别靠**结构不变量**，不靠**动态数据**
- 内容家族和点评家族底部基础结构相同 → 必须用 `business_line` 辅助区分
- 动态数据（种草数 / 观看量 / 身份标签）降级为 annotations「激活信号」

详见 [`donts.md`](./donts.md) 第 2 条 + [`experience.md`](./experience.md) 原则 8。

### 测试验证（v0.5.2 修正前后对比）

| 卡 | v0.5.1 识别（错）| v0.5.2 识别（对）|
| --- | --- | --- |
| Card 1（烤鸭控）| 内容家族（理由：无种草数）| 点评家族（种草数为 0，annotations.has_seed_count=false）|
| Card 4（酒店）| 点评 + "图偏横"误报差异 | 点评（删除"图偏横"误报）|

差异观察从 3 条降到 2 条（删除了内容层伪差异）。

## v0.5.1 (2026-04-14) · 五大家族体系贯穿全 Skill

把五大家族体系和 L1/L2 分层贯穿到 Skill 全身，所有文件互相呼应。

### 一致性体现

- `business-baselines.md` = 真相源（五大家族 + 22 基线 + 9 条规约）
- `layout.md` = 承载 L1
- `component.md` = 承载内容家族 L2
- `site-map.md` = 承载协议（business_line 枚举 + card_type 决策树）
- `SKILL.md` = 入口（架构 + 工作流）
- `prompts/page-level-review.md` = 执行契约

### 新增第 1 原则

**家族识别先于规则比对**——同一条规则应用到不同家族可能是不同合法值（"头像"在内容/点评家族必选，圈子/商品/新品家族不展示）。

## v0.5.0 (2026-04-14) · 页面级审查执行契约（agent 友好化）

Shaka 明确：**「之后 skills 我将给到其他 agent 去执行，所以要让他们能更好地理解」**——Skill 文档从「给设计师看的指南」升级为「**给 AI agent 执行的契约**」。

### 关键设计

- **Contract 而非指南**：输入/输出协议明确（yaml schema）
- **决策树用伪代码**：`IF...ELSE IF...` 可直接执行
- **每步自包含**：Step 0-6，每步输入/判断/输出都写死
- **兜底路径明确**：识别失败 / 业务线冲突 / 非本 Skill 范围 全部给规则

### 工作流三分化

| 工作流 | 入口 | 用例 |
| --- | --- | --- |
| 页面级审查（主流）| `prompts/page-level-review.md` | 生产巡检 |
| 单卡快检 | `prompts/review-screenshots.md` | 单点 / Debug |
| 变体评估 | `prompts/evaluate-variant.md` | 新变体评审 |

## v0.4.1 (2026-04-14) · 元信息位规约精细化

收口确认后追加点评·身份标签变体，暴露 v0.4.0 元信息位规约不够精细。

### 新洞察

v0.4.0 把「L1-3 标题下方 = 元信息位」当成单一位置。v0.4.1 修正为**两个位置按主语分**：
- 内容元信息 → 标题下方（紧贴内容描述）
- 人元信息 → 昵称旁（紧贴人）★ **新增 L1-9**

> **设计哲学沉淀**：「元信息按主语分位置，位置 = 主语的视觉锚点」——对应用户视线扫描的自然路径。详见 [`visual.md`](./visual.md) 第二节。

## v0.4.0 (2026-04-14) · 五大家族体系建立 + 22 基线沉淀 ⭐

Shaka 提供 22 张代表性卡片样式，逐张标注业务线/家族归属，完成**架构级升级**。

### 关键认知突破

- 京东双列卡不是二元（内容 vs 商品）而是 **五元家族**：内容 / 商品 / 圈子 / 点评 / 新品
- 每个家族有自己的**底部不变量（DNA）**
- 「**家族身份由底部决定，不由封面决定**」
- 「**顶部+中部统一识别语言、底部锚定家族身份**」= 京东双列卡的设计哲学

### 沉淀产物

- 22 个基线（5 家族子类型）
- 8 条跨家族规约（L1 大规范候选；v0.4.1 追加 L1-9 后达 9 条）
- 家族识别决策树
- 反哺其他文件清单

### 关键设计哲学

- 「**家族身份由底部决定**」——翻转对照（沉浸式·达人 vs 沉浸式·商品）的实证
- 「**新品家族 5 子类型 = 购买决策 5 阶段**」——造势 → 参与 → 验证 → 探索 → 共鸣
- 「**视觉权重 ∝ 信号公共性**」——公共信号强视觉，个人化信号弱视觉

## v0.3.0 (2026-04-14) · 定位调整 · 审查 → 观察+评价 ⭐

> Shaka 关键指示：「你发现不一致的地方不一定要纠错，你会给报告，告诉我们哪些不一样的，并用最开始教你的体验设计的视角去评价一下这样做的问题」。

触发**两个结构性调整**：

### 1. 定位反转

| 旧定位 | 新定位 |
| --- | --- |
| 列违规项，判 P0/P1/P2 | 列**差异观察** + 体验视角评价 |
| "你错了请修复" | "这里不一样；可能有 X 问题，也可能有 Y 业务合理性——请讨论" |
| 审查员 | 观察者 + 顾问 |
| 输出 = 判决 | 输出 = 讨论的抓手 |

### 2. L1/L2 规则分层

- L1 = 栅格、间距、圆角、字号值域、颜色 → **所有卡都走**
- L2 = 信息层级、头像、叙事 → **按 card_type 分支**

### 3. 双标签协议

- `card_type`（Skill 视觉识别）
- `business_line`（人工显式声明）

## v0.2.0 (2026-04-14) · 首次真实审查驱动的修订

首次真实审查（3 张截图：点评-求真官 / JoyAI-推荐 / JoyAI-直播）暴露了 v0.1 的结构性问题。

### 关键修正

- **新增原则 5**：流级别信息类型一致性（点评-求真官流里混了"店铺种草"和"UGC 测评"两种叙事）
- **输入协议铁律**：业务线归属必须人工显式声明，Skill 不推断
- **作者头像从"可选"改为"必选"**（v0.1 起草错误）
- **直播卡独立分支**（不能套图文卡规则）

## 演进的元规律

回看 v0.1 → v0.5.2，有几条规律值得未来其他 horizontal 专项 skill 借鉴：

| 规律 | 例证 |
| --- | --- |
| **真实测试驱动原则** | v0.2 / v0.5.2 都是实战测试发现的原则级问题 |
| **架构跃迁来自基线采集** | v0.4 五大家族体系来自 22 个基线的横向归纳 |
| **从「给人看」到「给 agent 吃」** | v0.5.0 把指南升级为执行契约 |
| **报告精简比报告全面更难** | v0.3.1 反向修订——少即是多 |
| **定位反转比规则修订更重要** | v0.3.0「审查→观察」是骨子里的转变 |

> 这些元规律值得纳入 Zone 5 horizontal 的方法论，作为后续专项 skill 的参考。当前 [`../README.md`](../README.md) 仅是 horizontal 总入口，元规律的系统沉淀属待办。
