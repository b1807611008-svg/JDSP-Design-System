# 体验判断 · Experience

> 8 条核心原则 + 体验判断维度工具箱 + 变体评估协议。
>
> **本文件给设计师 / 评审委员会看**——理解 Skill 的判断逻辑和评审标准；不是 Skill 的执行算法（执行算法在 Skill 仓库 `prompts/page-level-review.md`）。

---

## 一、8 条核心原则（按重要性排序）

### 原则 1：家族识别先于规则比对 ★ v0.5 新增

**Skill 必须先识别家族，再套规则**——按「**底部 → 封面 → 左上角**」顺序识别。

同一条规则应用到不同家族可能是不同的合法值（比如「头像」在内容/点评家族必选，在圈子/商品/新品家族不展示）。**先判家族，规则才有意义**。

### 原则 2：偏离即差异观察（不是违规判决）⭐ 定位锚 · v0.3.0 重写

对照规约识别差异。**不说"违规"，说"差异"+"体验代价"**。

每个差异配 4 个字段：
- **差异描述**（描述客观差异）
- **体验视角评价**（从维度工具箱选 1 个最贴切的讲透）
- **可能的业务合理理由**（辩证视角）
- **讨论建议**（推动设计师讨论）

> 这条原则定义了 Skill 的输出形态——**报告是讨论抓手，不是判决书**。

### 原则 3：跨场景同物种识别 · 变体评价核心 · v0.1 起

> **一张双列卡从场景 A 搬到场景 B，用户能否在 0.5 秒内无障碍识别它是同一物种？**
> - 能 → 通过（即便有差异）
> - 不能 → 认知负担，需讨论

**这是变体评价的一票否决标准**。详见下文「四、变体评估协议」。

### 原则 4：创新的举证责任倒置 · v0.1 起

**标准卡默认通过**。**创新卡 / 变体需要举证它创造了对等以上的用户价值**——而不是反过来要求审查方证明它有问题。

### 原则 5：流级信息类型一致性 · v0.2 新增（实战驱动）

**同一流的所有卡最好属于同一信息类型家族**。

视觉合规只是底线，**叙事一致性才是体验**。

> 实战例：某点评流里混了「店铺种草」+「UGC 测评」两种叙事，每张卡都合规但整体碎片化——这条原则就是为了发现这类问题（v0.2 真实审查驱动新增）。

### 原则 6：识别深度匹配规范深度 · v0.4 新增

**Skill 不做比规范更细的判定**。

业务方没规范的地方（梯度叠加每层、明星同款的具体趋势元素等），Skill 不下钻。Skill 的判定深度永远 ≤ 规范的明确深度。

### 原则 7：结构优先、内容无涉 ★ v0.5.2 新增（血教训）

**Skill 只观察设计师规范可控的结构性元素，不观察用户/运营上传的内容层**。

| 观察范围（Structure） | 不观察（Content）|
| --- | --- |
| 框体尺寸、圆角、间距、栅格 | 封面图具体内容、构图、比例偏向 |
| 字号、字重、颜色 token | 标题具体文案（只看字数合规性）|
| 组件有无、元素位置、层级结构 | 头像具体图像、运营配图色彩 |

**理由**：用户上传 / 运营配图是设计师无法强控制的内容层。规范只能约束「框」，不能约束「画里的内容」。

**例外**：图片比例**明显超出规范值域**（比如 1:2 极端比例）属规范违规要报；同一值域内不同选择（3:4 vs 4:3）不属于差异。

> 详细血教训见 [`CHANGELOG.md`](./CHANGELOG.md) v0.5.2 节和 [`donts.md`](./donts.md)。

### 原则 8：动态数据不作家族识别硬条件 ★ v0.5.2 新增（血教训）

**Skill 识别家族时，不能把"运行时数据"当成"结构标识"**。

**典型反例（v0.5.2 前的错误）**：
- ❌ 错：把"标题下方有种草数"当成点评家族的**必要条件**
- ✅ 对：种草数是动态数据（0 人种草就不显示），**不是结构标识**

**正确逻辑**：
- 家族识别靠**结构不变量**（固定元素的存在与位置）
- 动态数据（种草数、观看量等）作为**激活信号**记录在 annotations，**不参与家族判定**

## 二、体验判断维度工具箱

> Skill 给出每条差异时，从下述工具箱中选**一个最贴切的维度**讲透——不要每个都堆砌。

| # | 维度 | 适用场景 |
| --- | --- | --- |
| 1 | 身份识别 | 涉及用户认证 / 角色标签 / 创作者类型 |
| 2 | 认知负担 | 元素过多 / 信息冗余 / 同时呈现多种叙事 |
| 3 | 学习成本 | 与用户跨场景预期不符 / 需要重新学习 |
| 4 | 视觉一致性 | 同流卡之间样式跳跃 / 字号字重不齐 |
| 5 | 信息层级 | 视觉权重与信号公共性不匹配 |
| 6 | 社交信任 | UGC / 互动数据 / 公共社交证明的呈现 |
| 7 | 跨场景一致性 | 同物种在 A 场景能识别，到 B 场景识别成本骤增 |
| 8 | 行动号召 | CTA 按钮 / 进入路径的清晰度 |
| 9 | 视觉权重 | 公共信号 vs 个人化信号的视觉强度配比 |
| 10 | 叙事一致性 | 流级信息类型混杂 / 家族跳跃 |

**纪律**：每条差异**只挑一个维度**讲透。10 个维度堆叠 = 看不出哪个是核心问题。

## 三、报告输出形态（三段式）

> 详细模板见 Skill 仓库 [`prompts/page-level-review.md`](https://github.com/ShuaiMXu/jd-double-column-card-skill/blob/main/prompts/page-level-review.md) 报告模板节。

```
段 1 · 整体印象（≤ 3 句）
段 2 · 差异观察表（≤ 8 行；每格限字）
段 3 · 下一步建议（讨论方向，不是判决）
```

**输出纪律**（v0.3.1 精简后）：
- 差异表最多 8 行
- 每格限字（差异 ≤30 字 / 体验代价 ≤40 字 / 业务理由 ≤25 字 / 建议 ≤30 字）
- 整体印象最多 3 句
- 不按截图分节，按差异本身聚合
- 目标：**一屏能看完**

> 这个精简纪律是 Shaka 实战驱动的反馈：「报告简单点，刚看了一下很复杂呢」（v0.3.1）。原来八段式 + 10 维工具箱堆砌是过度设计。

## 四、变体评估协议（100 分制 + 豁免）

> 完整协议见 Skill 仓库 [`references/variant-evaluation.md`](https://github.com/ShuaiMXu/jd-double-column-card-skill/blob/main/references/variant-evaluation.md)。

### 4.1 评分维度（参考，不是判决）

100 分制由 4 部分组成：
- **跨场景同物种识别**（一票否决，原则 3）
- **结构合规性**（L1 + L2 规约满足度）
- **创新价值**（举证责任倒置，原则 4）
- **流级信息类型一致性**（原则 5）

> ⚠️ **打分降级为参考**（v0.3.0 调整）：分数只是数字化呈现，不再作为判决依据。最终决策在评审委员会。

### 4.2 豁免流程

变体可申请豁免，但必须：

| 规则 | 约束 |
| --- | --- |
| 同时活跃豁免数 | ≤ 5 个 |
| 豁免期 | 默认 ≤ 1 个迭代周期 |
| 期满处置 | 必须复盘，按数据决定**转正 / 回收 / 续期** |

### 4.3 当前活跃豁免

**当前无活跃豁免**（详见 Skill 仓库 `references/changelog.md` 豁免清单）。

## 五、给评审委员会的建议（解读 Skill 报告）

| 报告内容 | 评审委员会的处置原则 |
| --- | --- |
| Skill 标「差异 + 体验代价大」 | 需要业务线举证业务合理性 |
| Skill 标「差异 + 业务理由充分」 | 走豁免流程评估 |
| Skill 标「差异 + 业务理由空白」 | 默认要求合规 |
| Skill 标「无法识别家族」 | 业务线先确认 `business_line`，重跑 |
| Skill 标「不在观察范围」（内容层差异）| 不进入评审议程 |

**最终裁决权在评审委员会**——Skill 是辅助，不是替代。
